# CLAUDE.md

Orientação perene para contribuidores (humanos ou agentes). Mantenha este arquivo atualizado quando decisões fundamentais mudarem.

## Purpose

`nengine` é uma 2D game engine **construída para aprender arquitetura de engine** — clareza didática acima de performance prematura, evolução incremental guiada por jogos de exemplo. Stack: Kotlin + Skiko (backend de render padrão, `:engine-skiko`) com LWJGL (`:engine-lwjgl` sobre NanoVG + GLFW + OpenGL 3.3 core) como segundo backend ativo. Scene graph estilo Godot — comportamento por herança, sem ECS. Quickstart, capacidades e demonstrações vivem em [`README.md`](./README.md); este arquivo concentra invariantes, convenções e mapa de pastas.

## Architectural Invariants

Toda mudança deve respeitar os seis invariantes abaixo. Eles vêm das decisões arquiteturais consolidadas no histórico em `openspec/changes/archive/` e não podem ser quebrados sem uma nova change OpenSpec discutindo a revisão.

1. **Scene graph estilo Godot, por herança.** Comportamento de gameplay é adicionado por subclasses de `Node` / `Node2D`. **Sem** `List<Component>` ou ECS. Cada Node tem sua identidade de tipo (`class Paddle : Node2D()`).

2. **`:engine` não depende de nenhum framework de UI/render.** O módulo `:engine` não declara — direta ou transitivamente — `org.jetbrains.compose.*`, `androidx.compose.*`, `org.jetbrains.skia.*`, `org.jetbrains.skiko.*`, `org.lwjgl.*`, AWT/Swing além do estritamente necessário, nem futuras dependências OpenGL/Vulkan/WebGPU. Quem precisa de qualquer um desses é o módulo de backend correspondente.

3. **Colisão via `CollisionObject2D` + `CollisionShape2D` + `PhysicsSystem` central.** Cada participante de colisão é uma subclasse de `CollisionObject2D` (`Area2D` para triggers; `StaticBody2D`/`CharacterBody2D`/`RigidBody2D` para corpos sólidos) com um ou mais `CollisionShape2D` filhos carregando um `Shape2D` polimórfico (`RectangleShape2D` ou `CircleShape2D`). `PhysicsSystem.step(tree, dt)` integra `RigidBody2D`, executa o TOI loop com impulse solver bilateral (linear + angular + Coulomb tangencial), e dispara `onAreaEntered`/`onAreaExited`/`onBodyEntered`/`onBodyExited` exatamente uma vez por par-transição (enter no início da sobreposição, exit no fim — não há evento "still touching"). Os mesmos eventos são emitidos via signals built-in (`areaEntered`, `areaExited`, `bodyEntered`, `bodyExited`) em cada object. `Area2D` permanece sensor puro (sem `moveAndCollide`); para "quem está em mim agora?" use `Area2D.getOverlappingAreas()` / `getOverlappingBodies()`.

   ### RigidBody2D vs CharacterBody2D

   A engine oferece **dois modelos de movimento solid-body** com critérios claros:

   - **`CharacterBody2D`** — controle direto pelo script. O script lê input, computa `velocity`, e chama `moveAndCollide(velocity * dt)` para mover com sweep CCD-correto. **Sem resposta automática**: sem massa, sem restituição, sem inércia. Use para entidades sob controle do jogador ou IA simples (paddles em Pong, snake head, player platformer, naves em Asteroids).
   - **`RigidBody2D`** — integração pela engine. Script aplica forças/impulsos (`applyForce`, `applyImpulse`, `applyTorque`, etc.) ou seta velocidades diretamente; a engine integra, faz sweep com TOI loop e resolve cada contato via impulso linear+angular + fricção de Coulomb. Use para objetos "soltos no mundo" com semântica física real (bolinhas quicando, blocos empurrados, projéteis com gravidade, peças de demolição).

   Em colisão cruzada, `Static` e `Character` são tratados como massa infinita pela perspectiva do `Rigid` — o `Rigid` recebe impulso, eles não recoluem (preserva o controle direto pelo script). Para "kick" do Character no Rigid, o script chama `rigid.applyImpulse(...)` manualmente dentro de `_on_body_entered`.

4. **`Renderer`, `Input` e `GameHost` permanecem SPIs obrigatórias.** Skiko é o backend padrão (`:engine-skiko`), usado por todos os jogos shipped; `:engine` não pode vazar tipos de Skiko (`org.jetbrains.skia.*`, `SkikoView`, `SkiaLayer`). LWJGL é o segundo backend ativo (`:engine-lwjgl`, sobre NanoVG + GLFW + OpenGL 3.3 core via `org.lwjgl.*`), exercido pelo entrypoint LWJGL de `:games:demos` como sentinela viva deste invariante. `:engine` também não pode vazar tipos de LWJGL — quem precisa dessas bindings é o `:engine-lwjgl`. Jogos novos devem usar Skiko por default. **`GameHost.render` não desenha** — toda saída visual passa por `SceneTree.render`. O host também **não toca em `tree.debug.*` por frame**, não instancia `FpsCounter`, não chama nenhum helper de overlay e não polla teclas para toggle de visualização: a única referência permitida a debug é a atribuição única `tree.debugHudKey = config.debugHudKey` no startup. O polling do `debugHudKey` vive num `Node` interno (`DebugToggleNode`) dentro da `DebugLayer` auto-inserida pela engine.

5. **A árvore viva é dona de `SceneTree`, não de uma `Scene` que é Node.** `SceneTree` não é `Node` e não é `@Serializable`; a classe `Scene` não existe mais em `:engine`. Nodes alcançam a árvore via `node.tree` (set no attach, null no detach). `SceneTree` não é subclassável para customizar setup — para popular a árvore inicial, escreva um Node root com `onEnter`. `SceneLoader.load` e `BundleLoader` devolvem `Node` (root livre); o host envolve em `SceneTree(root = ...)` antes de `run(...)`.
6. **UI in-game vive em screen-space via `CanvasLayer`, esticada em design-space.** `CanvasLayer` é `Node` (não `Node2D`) e estabelece um sub-pass de render que **ignora a view transform da `Camera2D` corrente**. Carrega `followStretch: Boolean` (`@Inspect`, default `true`): quando ligado, seus descendentes `Node2D` são resolvidos, desenhados e hit-testados em **design-space** (`tree.designSize`) e a engine empilha uma **UI stretch transform** que mapeia o design rect na superfície (letterbox), fazendo posição, tamanho e `fontSize` escalarem em sincronia com o mundo no resize; quando `false`, ficam em pixels de tela crus (`tree.size`) — o comportamento pré-stretch, usado por overlays pixel-locked (o `ScreenDebugCanvas` é `followStretch = false`). A `SceneTree` é dona da resolução de design: `designSize: Vec2` e `uiStretchMode` (`UiStretchMode { FIT, FILL, STRETCH, DISABLED }`, default `FIT`). `designSize` é defaultado **uma vez** no `start()` a partir do `bounds` da `Camera2D` corrente (então é **estável** — pan/zoom da câmera não perturbam o HUD), caindo para a superfície quando não há câmera (nesse caso **rastreia** a superfície no resize, mantendo o stretch identidade / raw screen-space); um write explícito congela o valor. `uiStretchTransform()` computa `(translation, scale)` via `fitTransform(designSize, size, mode)` — só a adaptação de resolução (escala + centralização do letterbox), **sem** o pan da câmera — e é `null` (sem push) em `DISABLED`/degenerado/identidade; `Camera2D.computeViewTransform` reusa o mesmo `fitTransform` e soma o termo de pan (`−bounds.origin·scale`) por cima, garantindo que as barras de letterbox de mundo e HUD coincidam. `SceneTree.render` executa dois passes: (a) world pass em DFS pulando subárvores de `CanvasLayer`, com a view transform do `Camera2D` atual aplicada; (b) UI pass enumerando `CanvasLayer`s em `(layer asc, dfs-order asc)` — `layer` é `Int`, maior pinta na frente — empilhando a UI stretch transform em volta de cada layer `followStretch` (layers crus partem da identidade); o `hitTestUI` mapeia o ponteiro de volta ao design-space (inverso do stretch) antes de testar `Button`s de layers esticados. `Control` (`abstract class Control : Node2D`, em `com.neoutils.engine.scene`) é a base Godot-style dos widgets de UI: `Panel`, `Button`, `Label` e `ColorRect` estendem `Control` (que sobe o `size: Vec2`, `localBounds = Rect(ZERO, size)`, e expõe anchors/offsets, `visible` e `mouseFilter`); `Circle2D` permanece `Node2D` puro (forma centrada, não retângulo ancorável). `Label` é um caso **min-size** — seu rect é o texto medido (`fontSize` é a altura da fonte, renomeada do antigo `size`), sem `size` de rect setável. Anchors/offsets são a fonte de verdade Godot 4-style (`edge = anchor*parentDim + offset`); um **anchor layout pass** em `SceneTree` (`runAnchorLayout`, top-down, parent rect = ancestral `Control` mais próximo ∨, na fronteira de um `CanvasLayer`, o design rect `Rect(ZERO, tree.designSize)` quando `followStretch` ∨ a superfície `Rect(ZERO, tree.size)` quando cru) resolve `position`/`size` de cada `Control` **antes** do UI render pass e do `hitTestUI` a cada tick, matando o relayout-por-frame; escrever `position`/`size` faz write-back nos offsets (compat com cenas que só setam `position`/`size`). `visible = false` pula o `Control` e sua subárvore no render **e** no hit-test; `mouseFilter` (`STOP`/`PASS`/`IGNORE`) decide consumo do clique (default `STOP` em `Panel`/`Button`, `IGNORE` em `Label`/`ColorRect`). Os campos `focusMode`/`focusNeighbor*`/`sizeFlags*` nascem declarados mas **inertes** (acesos por `ui-focus`/`ui-layout`). `Panel` (retângulo + border opcional) e `Button` (signal `pressed` built-in, hover/press/disabled visuals, hit-test geométrico via `screenRect()`) seguem como folhas UI shipped — qualquer Node2D pode viver dentro de um `CanvasLayer`, sem demanda especial. A engine auto-insere uma `DebugLayer` (`Node`, `name = "__debug"`) como filho do root contendo dois sub-containers: `WorldDebugContainer` (`Node2D` direto, participa do world pass com a view transform da `Camera2D`) e `ScreenDebugCanvas` (`CanvasLayer` em `layer = Int.MAX_VALUE - 1`). A engine roteia widgets registrados via `tree.debug.register(...)` pelo subtipo: `WorldDebugWidget` vai no world container, `ScreenDebugWidget` vai na screen canvas. `tree.debug` é um `DebugRegistry` per-tree com fields `colliders` / `log` / `hud` (+ os gizmos de física e o `profiler`) para os built-ins. O `colliders` carrega um modo `ColliderDrawMode { AABB, REAL }` (default `REAL`) que funde o antigo `ShapeGizmoWidget` (geometria real) com o envelope de broad-phase; o modo é escolhido por um painel screen-space companheiro (`ColliderModePanel`, segmented control `AABB | REAL`) que espelha `colliders.enabled` e vive fora do HUD (braço screen-space do colliders, no espírito do split mestre/escravo do Inspector). O Inspector é a ferramenta de depuração de scene graph: o mestre `inspector` (`SceneTreeWidget`) é a view tree navegável, dona da seleção (`selected` + `select`) e a única row na HUD; os escravos `nodeInspector` (`NodeInspectorWidget`, painel de detalhe screen-space) e `selectionGizmo` (`SelectionGizmoWidget`, gizmo world-space) derivam o `enabled` do mestre e ficam fora de `widgets`/HUD; o pick por click no mundo entra via `SceneTree.hitTestPick`. Fechar a view tree (`[x]`) desliga toda a ferramenta. O `profiler` embute a leitura de `fps` (amostrada via `nanoTime`, independente da instrumentação de fases). Não há mais widgets/fields `fps`, `momentum` ou `shapeGizmo`. `SceneTree.hitTestUI(input)` corre antes de `tree.process(dt)` no `GameLoop.tick`; um `Control` `STOP` (ou `Button` armado) consumido seta `input.mouseClickConsumed = true`, e `Input.wasMouseClicked(Left)` passa a retornar `false` (use `wasMouseClickedRaw` para ver o bruto). Containers (HBox/VBox), focus, theme e widget set estendido vivem em changes futuras (`ui-layout`, `ui-focus`, `ui-theme`, `ui-input-events`) — `ui-anchors` foi **absorvida** por `ui-controls-base`, e `ui-focus`/`ui-layout` apenas **acendem** campos (`focusMode`/`sizeFlags`) que `Control` já declara.

## Module Layout

| Módulo                  | Responsabilidade                                                                                                                                                                                                  |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `:engine`               | núcleo Kotlin puro (scene graph, math, SPIs, física, loop, debug widgets/HUD em `com.neoutils.engine.debug.*`, logging em `com.neoutils.engine.dx.*`, `GameHost` SPI)                                              |
| `:engine-bundle`        | carregar cena via bundle (scene.json + scripts/) e hospedar a SPI `ScriptHost` agnóstica de linguagem                                                                                                              |
| `:engine-bundle-python` | implementação concreta de `ScriptHost` para scripts Python `.py`, usando GraalPy 24.x; publica stubs `.pyi` em resources                                                                                           |
| `:engine-bundle-lua`    | implementação concreta de `ScriptHost` para scripts Lua `.lua`, usando LuaJ 3.0.x (JAR puro JVM); publica stubs LuaCATS em resources                                                                               |
| `:engine-skiko`         | implementação de `Renderer`/`Input`/`GameHost` SPIs via Skiko + SkiaLayer + JFrame; backend padrão usado por todos os jogos shipped                                                                                |
| `:engine-lwjgl`         | implementação de `Renderer`/`Input`/`GameHost` SPIs via LWJGL (NanoVG + GLFW + OpenGL 3.3 core); segundo backend ativo de render; sentinela do invariante #4 via entrypoint LWJGL de `:games:demos`. Único módulo do projeto autorizado a declarar dependência em `org.lwjgl.*`. |
| `:engine-audio-javasound` | implementação de `AudioBackend` (`JavaSoundAudio`) em JDK puro (`javax.sound.sampled`), host-agnóstica — o mesmo módulo serve `:engine-skiko` e `:engine-lwjgl`. WAV-only no v1, vozes sobrepostas (uma linha nova por `play`), resolução de asset por classpath. Depende só de `:engine`; nenhuma lib de áudio de terceiro. |

Os módulos `:shared` e `:desktopApp` do template KMP foram **removidos** durante a change `engine-foundation`.

**SPI de áudio (server-style).** Saída de som é um serviço de tree, não um Node/Component (invariante #1): `SceneTree.audio: AudioBackend?` é nullable e injetado pelo host no startup, espelhando exatamente `tree.textMeasurer`. Nodes/scripts disparam SFX off-frame via `node.tree.audio?.play(sound)` — `null` ⇒ no-op gracioso (testes/headless silenciosos). A SPI (`AudioBackend` + handle opaco `Sound`) vive em `com.neoutils.engine.audio` (`:engine`, Kotlin puro, sem `javax.sound`/binding nativo — invariante #2); a impl concreta mora em `:engine-audio-javasound`. `tree.stop()` chama `audio?.dispose()` uma vez no teardown. v1 é SFX curto fire-and-forget WAV-only; música/posicional ficam para changes futuras (a SPI `load`/`play` não muda quando entrarem).

**SPI de textura (server-style).** Carregar imagem é serviço de tree, não Node/Component (invariante #1): `SceneTree.textures: TextureBackend?` é nullable e injetado pelo host no startup, espelhando `tree.audio`/`tree.textMeasurer`. Nodes resolvem assets via `node.tree.textures?.load(path)` — `null` ⇒ no-op gracioso (sprite invisível em headless/teste, sem erro). `load(path)` resolve por classpath, decodifica **uma vez** e **cacheia por path** (mesmo path ⇒ mesmo handle); fail-fast em ausente/ilegível. A SPI (`TextureBackend` + handle `Texture`, que expõe **só** `width`/`height` — nenhum byte/pixel/tipo de backend vaza) vive em `com.neoutils.engine.render` (`:engine`, Kotlin puro, sem `org.jetbrains.skia.*`/`org.lwjgl.*` — invariante #2). Diferente do áudio, **o backend é por módulo de render** (uma textura É o objeto do pipeline gráfico): `SkikoTextureBackend` (`Texture` = `org.jetbrains.skia.Image`) em `:engine-skiko`, `LwjglTextureBackend` (imagem NanoVG) em `:engine-lwjgl`. `tree.stop()` chama `textures?.dispose()` uma vez no teardown. O desenho é per-frame e passa pela transform stack como qualquer primitiva: `Renderer.drawImage(texture, src, dst, flipH)` — `src` em px da textura, `dst` em local space, amostragem **nearest-neighbor** (pixel-art não pode borrar), `flipH` espelha visualmente sem `scale.x` negativo. O node `Sprite2D : Node2D` resolve `texturePath` no `onEnter` e desenha a textura centrada na origem local. Sobre essa fundação, `AnimatedSprite2D : Node2D` cicla um **sheet horizontal** de `frameCount` quadros (largura do quadro = `texture.width / frameCount`, altura = `texture.height`): a engine avança `currentFrame` no `onProcess` a `fps` quadros/s (`loop` faz wrap; sem `loop` satura no último e zera `playing`), e o `onDraw` reusa `drawImage` recortando o quadro corrente — sem nova op de `Renderer`. Ainda sobre a mesma fundação, `TileMap : Node2D` desenha uma **grade de tiles** a partir de um `TileSet` (descritor `@Serializable` não-Node, análogo a `Shape2D`: `texturePath` + `tileWidth`/`tileHeight`, com `columns = texture.width / tileWidth` derivado do atlas e `src(index)` mapeando índice→célula row-major do atlas). O `TileMap` carrega `columns`/`rows`/`tiles: List<Int>` (row-major, `-1` = célula vazia); o `onDraw` reusa `drawImage` uma vez por célula não-vazia, com origem no canto superior-esquerdo da grade (não centra, diferente de `Sprite2D`). **Tilemap é visual-only**: `TileMap` **não** é/cria `CollisionObject2D`, não carrega `CollisionShape2D` e não participa do `PhysicsSystem` — colisão de terreno é feita à mão com `StaticBody2D` posicionados separadamente (invariante #3 intacto). v1 é render estático/animação por sheet 1D/tilemap 1-camada visual PNG-only; catálogo nomeado (`SpriteFrames`)/atlas genérico/tint, múltiplas camadas de tilemap, autotiling e geração de colisão a partir de tiles ficam para changes futuras (a SPI não muda quando entrarem).

## Coding Conventions

- **Comentários só para o "por quê" não-óbvio.** Nunca para o "o quê" (nomes já dizem). Evite docstrings cerimoniais.
- **Identificadores em inglês.** Texto in-game e mensagens ao usuário podem ser em português; nomes de classe/função/variável devem permanecer em inglês.
- **API pública de `:engine` documentada com KDoc** quando o uso pretendido não for óbvio.
- **Imutabilidade onde for barata.** `Vec2`, `Rect`, `Transform`, `Color` são data classes; operações retornam novas instâncias. `Node2D` expõe `position: Vec2`, `rotation: Float` e `scale: Vec2` como properties (açúcar sobre `transform = transform.copy(...)`); escrever componente individual (`vec2.y = X`) é proibido por design e falha em runtime para scripts.
- **Folhas `Node2D` shipped por `:engine` são `open` por default.** Coerência com o invariante #1 ("comportamento adicionado por herança"). Tornar uma folha `final` exige justificativa documentada no KDoc explicando que invariante quebraria sob herança.
- **Sem dependências escondidas.** Se um módulo precisa de um framework de UI/render (Skiko, AWT/Swing, LWJGL), declara no `build.gradle.kts` daquele módulo. Dependência transitiva proibida em `:engine` é bug.
- **Testes para regras invariantes.** Cada decisão arquitetural com risco de regressão (lifecycle ordering, broad phase, cache de transform) tem teste unitário.
- **`Node2D.onDraw` desenha em local space.** Cada subclasse de `Node2D` (shipped ou de jogo) implementa `onDraw` em coordenadas **locais** — origem `(0, 0)`, sem somar `world().position` manualmente. `SceneTree.render` empilha a `transform` local via `Renderer.pushTransform` antes do `onDraw` e desempilha depois; rotação e escala dos ancestrais chegam ao desenho pela stack do renderer.
- **Disciplina `@Inspect` / `@Transient` para classes `@Serializable`.** Toda `var` em classe `@Serializable` é anotada **ou** com `@Inspect` (configuração inicial: persiste em scene file, exposta ao editor futuro) **ou** com `@Transient` do `kotlinx.serialization` (estado runtime: nunca persiste). Não deixe uma `var` sem anotação. `val` não precisa de anotação.

## Scripting Model

A engine suporta Python (via GraalPy em `:engine-bundle-python`) e Lua (via LuaJ em `:engine-bundle-lua`) como mecanismos de scripting. O modelo é Godot-style: hooks underscore-prefixed (`_ready`, `_process`, `_physics_process`, `_draw`, `_exit_tree`, `_on_area_entered`/`_exited`, `_on_body_entered`/`_exited`), exports descobertos estaticamente, signals declarados via factory `signal(...)`, fail-fast em qualquer erro (parse, `extends` desconhecido, exceção num hook).

Contratos detalhados (estrutura do script, exports, scene.json properties bag, bindings de tipo, sandbox de `require` em Lua, signals built-in dos `CollisionObject2D`) vivem em:

- [`openspec/specs/python-scripting/`](./openspec/specs/python-scripting/spec.md)
- [`openspec/specs/lua-scripting/`](./openspec/specs/lua-scripting/spec.md)
- [`openspec/specs/scene-serialization/`](./openspec/specs/scene-serialization/spec.md) — properties bag e roteamento entre `@Inspect` e exports
- [`openspec/specs/bundle-loading/`](./openspec/specs/bundle-loading/spec.md) — pipeline de carga `BundleLoader.fromResources`

## OpenSpec Workflow

Mudanças materiais (arquitetura, API pública, novos módulos, novas capabilities) **passam por uma change OpenSpec antes da implementação**. Roteiro padrão:

1. Explore: `/opsx:explore <tema>` — gera notas e perguntas.
2. Propose: `/opsx:propose <change-name>` — gera proposal, design, specs, tasks.
3. Apply: `/opsx:apply <change-name>` — implementa tarefa por tarefa.
4. Verify: `/opsx:verify <change-name>` — confere se a implementação fecha com os artefatos.
5. Archive: `/opsx:archive <change-name>` — congela a change e atualiza specs principais.

Para uma feature nova ou refator significativo: abra uma change OpenSpec, **não** um PR direto de código.
